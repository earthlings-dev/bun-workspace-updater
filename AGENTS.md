# Repository Guidelines

## Project Structure & Module Organization

This is a small Bun + TypeScript utility. Source code lives in `src/`, centered on `src/update-latest.ts`, a CLI that updates package ranges in `package.json` files while preserving JSON formatting through TypeScript JSON AST positions. Tests live in `tests/` as a single `update-latest.test.ts`; user-facing docs are in `README.md`. Root configuration files include `package.json`, `bunfig.toml`, `tsconfig.json`, `tsconfig.strict.json` (the strict shared base that `tsconfig.json` extends), and `eslint.config.ts` (flat, type-aware ESLint config). `node_modules/` is generated and must not be edited by hand. `coverage/` is generated by the test runner but is **intentionally tracked** in this repo (it is deliberately not git-ignored) — do not re-add it to `.gitignore` and do not hand-edit its contents. There is no dedicated assets directory.

## Build, Test, and Development Commands

- `bun install`: install dependencies and update `bun.lock`.
- `bun run src/update-latest.ts --dry-run`: run the updater without writing files.
- `bun run src/update-latest.ts --tag next --only typescript`: test registry lookup options against selected packages.
- `bun run src/update-latest.ts --no-recursive`: limit the update to the root `package.json` (recursion is on by default whenever `workspaces` is declared; `-r`/`--recursive` force it on). Add `--resolve-sentinels` to also pin `latest`/`next`/`*`/`x` ranges.
- `bun run src/update-latest.ts --to-catalog --dry-run`: preview migrating array-form workspace dependencies into a Bun `catalog` (unifies version conflicts to the latest range).
- `bun run test`: run Bun tests (the `test` script is `bun test`) using settings from `bunfig.toml`, including coverage output in `coverage/`.
- `bun run typecheck`: type-check the project (runs the local `tsc --noEmit`) with the strict shared TypeScript config.
- `bun run lint` / `bun run lint:fix`: run ESLint (`eslint .` / `--fix`) with the strict, type-aware flat config. ESLint loads `eslint.config.ts` natively under Bun — no `jiti` — because `bunfig.toml`'s `[run] bun = true` runs the binary under Bun.

`package.json` defines five scripts: `lupdate` (`bun run src/update-latest.ts`), `test` (`bun test`), `typecheck` (`tsc --noEmit`), `lint` (`eslint .`), and `lint:fix` (`eslint . --fix`). Invoke tooling through `bun run <script>` — not `bunx` (which auto-downloads a non-pinned binary) and not the bare `bun test` subcommand.

## Coding Style & Naming Conventions

Use TypeScript modules with ESM imports. Keep strict type-safety intact: avoid `any`, handle `undefined`, and preserve readonly annotations where data is not mutated. Keep public surfaces explicitly typed; prefer explicit typed bindings over loose casts or inferred default exports. Follow the existing style in `src/update-latest.ts`: two-space indentation, **single quotes** (double quotes only inside emitted JSON, where the format requires them), semicolons, descriptive interfaces, and small pure helper functions before CLI orchestration. Use camelCase for variables and functions, PascalCase for interfaces and types, and uppercase snake case for constants such as `DEP_FIELDS`. This repo is Bun-first: do not introduce `node`, `npm`, `pnpm`, `yarn`, or compatibility layers unless the user explicitly asks for them (the one allowed Node import is `node:path`; tests also use `node:os` / `node:fs/promises` for temp dirs). Most of these conventions are enforced by ESLint (see Linting) — `bun run lint:fix` will normalize formatting — so prefer fixing the code over loosening a rule.

## Testing Guidelines

Use Bun’s test runner via `bun run test` (= `bun test`) — never `bunx` and never the bare `bun test` subcommand. Name tests with `*.test.ts`; files matching `*.concurrent.test.ts` run under the concurrent glob configured in `bunfig.toml`. Coverage thresholds are 95% for lines, functions, and statements, so new behavior should include focused tests for argument parsing, range filtering, AST collection, and edit application. Keep the suite in **one file** so coverage merges into one accurate report; check for existing helpers before adding new ones.

Mock the **specific real invocation**, not an injected wrapper. Pure-logic tests are `test.concurrent` (they touch no process-global state). Host-effect tests mock the actual call with `spyOn` at the global call site — `spyOn(Bun, 'write')`, `spyOn(Bun, 'spawn')`, `spyOn(globalThis, 'fetch')`, `spyOn(console, …)`. **This is the only seam that intercepts Bun's built-in APIs:** `src` calls the real globals (`Bun.write`/`Bun.spawn`/`fetch`/`console.*`); mocking the `bun` MODULE via `mock.module('bun', …)` — or spying a `bun` namespace import — does NOT intercept them (verified: `src` hits the real registry / runs the real `bun install`). So there is **no** `HostIo`/injection boundary; the global spy is the working mechanism. Build the spies with a **factory** (`installHostMocks()` returning the handles) and install/tear them down through a parent-`describe` **lifecycle**: `beforeEach(() => { h = installHostMocks(); })` + `afterEach(() => { mock.restore(); })` (scoped to the host-effect `describe`, so the `.concurrent` pure tests are untouched). Use **dynamic** mocks for varying behavior: `mockResolvedValueOnce(...)` chains for sequential calls (the `fetchLatest` unit tests), `mockImplementation(...)` for the URL-routed registry, `mockReturnValue(...)` to override the install exit. Because `spyOn` patches a process-global, host-effect tests run serially (plain `test`). Mock `Bun.spawn` to return a **real** `Bun.spawn(['true'])` / `['false']` subprocess so the install exit code is genuine without running `bun install`. Give each filesystem test its own `mkdtemp` dir built with `node:fs` (so setup never hits the mocked `Bun.write`) and read it back through real `Bun.Glob` / `Bun.file`. `main` **returns** an exit code (the `import.meta.main` guard owns `process.exit`). Assert with Bun’s deep-equality (`toEqual` / `toStrictEqual`) and the asymmetric `stringContaining` / `objectContaining` matchers; assert rejections via a `try`/`catch` helper (Bun types `expect().rejects.toThrow()` as `void`). `spyOn` patches the real binding in place, sidestepping the `Mock<T>` typing pitfalls (dropped `fetch.preconnect`, `Bun.write` / `Bun.spawn` overload incompatibility) that an injected typed double would hit. Do not create throwaway/disposable probe test files; fold any capability check into the real suite.

## Linting

ESLint is configured for strict, type-aware linting via a flat, shareable TypeScript config in `eslint.config.ts`, composed from each plugin's ESM shareable configs through `defineConfig` / `globalIgnores` (from `eslint/config`):

- `@eslint/js` — `js.configs.recommended`.
- `typescript-eslint` — `tseslint.configs.strictTypeChecked` for **strict, type-aware correctness only** (its base registers `tseslint.parser` + `tseslint.plugin`). typescript-eslint's own formatting/style rules were removed years ago in favor of `@stylistic`, so no style ruleset is layered from here (we do not use `stylisticTypeChecked`). Typed linting uses `parserOptions.projectService`, so every linted file must belong to a TS project: `src` and `eslint.config.ts` are in `tsconfig.json`, while `tests/` is not — so `tseslint.configs.disableTypeChecked` turns the type-aware rules off for tests (syntactic and `@stylistic` rules still apply).
- `@stylistic/eslint-plugin` — **the idiomatic TypeScript style ruleset** (formerly-core and formerly-typescript-eslint formatting rules now live here). Built from the **factory** `stylistic.configs.customize({ indent: 2, quotes: 'single', semi: true, jsx: false })`, aligned to the repo style (note `brace-style: stroustrup`, `operator-linebreak: before`, `arrow-parens: as-needed`). Use the factory, not `stylistic.configs.all`.

Run with `bun run lint` / `bun run lint:fix`. **No preset rules are disabled.** Where `@typescript-eslint/prefer-as-const` pressed on the repo's no-`as const` convention, the discriminant consts are written idiomatically as `const X = 'lit'` (a top-level `const` infers the literal type), which satisfies the rule without `as const` — see Discriminated-Union Variant Naming. Fix lint findings in the code; do not add `eslint-disable` comments or loosen rules without cause.

## Commit & Pull Request Guidelines

Commit subjects must use `type(scope): imperative description`, for example `docs(readme): document --to-catalog migration`, `feat(cli): add workspace recursion`, or `test(cli): cover captured Console output`. Choose a meaningful type; do not use lazy categories like `chore`. Commit bodies should contain 1-5 plaintext sections. Put each section header immediately before its first bullet list item, with no blank line between the header and bullets. Each section should have 3-5 bullet items, and sections should be separated by one blank line.

Pull requests should include a concise summary, the behavior change, and the commands run — typically `bun run typecheck`, `bun run lint`, and `bun run test`. Link related issues when available and include terminal output snippets for CLI-facing changes.

## Security & Configuration Tips

The updater fetches package metadata from the npm registry and may run `bun install`. Use `--dry-run` before writing changes and `--no-install` when reviewing edits manually. Do not commit registry tokens or local environment files. (Note: unlike the usual convention, `coverage/` **is** intentionally tracked in this repo — committing it is expected, not a mistake.)


## Bun Docs First

- Before searching the web for Bun behavior, APIs, testing, mocking, or file-system guidance, read the local Bun reference docs first.
- Primary local Bun reference: `~/bun-full-llms.md`
- Secondary local Bun reference: `~/bun-index-llms.md`
- Treat those files as the default source of truth for Bun-native and Bun-first implementation choices in this repo.
- If those files are more than a day old, run `cd ~ && fetchbun 2>&1;` to retrieve the latest versions.
- For module-relative filesystem roots, use Bun's direct module metadata: `import.meta.dir` or `import.meta.path`. Do not introduce handwritten "find the project root" or "walk upward until package.json" helpers when the code is operating from a known module boundary.
- For file discovery, prefer `Bun.Glob` with an explicit `cwd` root over handwritten recursive `fs.readdir*` walkers. If the workload or test helper is only discovering matching files, a manual directory walk is considered the wrong mechanism in this repo.
- Use real Bun public namespace types directly, such as `Bun.PathLike`, `Bun.BunFile`, `Bun.S3File`, `Bun.BlobPart`, `Bun.Archive`, and `Bun.SpawnOptions.*`. Do not recreate Bun host surfaces locally when the real namespace type is available.
- When a `typescript-eslint` parser-services type is needed, import it directly from `@typescript-eslint/typescript-estree`. Do not use non-exported subpaths, local facsimiles, or context-shaped wrappers to approximate parser services.
- Avoid helper guards that exist only because a local abstraction lost the direct upstream type. Prefer direct upstream unions and discriminants over repo-local pseudo-narrowing helpers.

## No Workarounds

- Do not "simplify first" when the upstream surface is strict, overloaded, or already public. Model the real surface directly.
- Do not introduce generic callback wrappers, umbrella selector helpers, pseudo-unions, "temporary" local facsimiles, or assertion-based recovery layers around Bun or `typescript-eslint` APIs.
- Do not carve repo-owned request, report, or engine contracts into smaller transport-only subset types such as `Serializable*`, `*Core`, `Pick<...>`, or omit-based facsimiles just to cross a worker/process boundary. If a boundary cannot carry the current contract, redesign the authoritative contract or move the transport concern into a separate observer/channel parameter.
- Progress, streaming, and host-write hooks are transport concerns, not part of a pure domain request. Do not bolt callback sinks onto request/report objects just to make a caller side-effectful; pass those hooks through a separate explicit boundary.
- If the exact upstream contract feels awkward, that is a signal to redesign the local implementation so the upstream type survives intact through the boundary, not a reason to widen, erase, or reconstruct the type later.
- Prefer real typed factories and real typed constructors over unannotated object literals that happen to "look right". A canonical constructor/factory is the scalable way to preserve optional fields, array element types, and future evolution.
- Do not use `as const`, broad `readonly string[]` annotations on literals, proxy result casts, or generic "selector" abstractions in this repo.
- Evidence matters here: when agents took shortcuts around exact upstream contracts, the repo produced cascades of `no-type-assertions`, `no-defensive-annotation-widening`, `exactOptionalPropertyTypes`, overload incompatibility, and top-level side-effect errors. Doing it the strict way reduced the error set instead of moving it around.
- If a shortcut appears to save time but changes the type shape, overload set, lifecycle model, or ownership model, assume it will create more issues downstream. Stop and implement the structurally correct version first.

## Discriminated-Union Variant Naming

When a discriminated union has real variants, each variant deserves a named type. The union exists for positions that genuinely accept "any of these" (polymorphic return types, polymorphic parameters, fields that hold values of multiple shapes). Construction sites annotate against the variant, not the union. Anonymous arms inside a union — `= | Readonly<{ kind: 'a'; ... }> | Readonly<{ kind: 'b'; ... }>` — are widening hazards at every construction site. Name each variant, then compose them into the union by name.

Widening and narrowing are tools the language gives you when the type-level structure doesn't already match the value-level reality. If every variant is a concrete named type, every literal is annotated against its specific variant, and every union is used only where polymorphism is genuine, neither widening nor narrowing has to happen.

The principle isn't "object variants get named, scalars don't" — it's shared variant ownership across the owner/caller boundary. The owner defines each variant as a named type; the caller imports that exact type and constructs the specific variant (annotated), so the value is born at its narrowest type. The failure mode I kept designing toward is the opposite: the caller builds an inline literal checked against the union, and correctness depends on structural matching + later narrowing. That's fragile precisely because owner and caller don't share a definition — they share a shape and hope. Naming every variant (scalar included) and constructing the specific one removes the narrow-after-the-fact step entirely.

If you are doing type widening/narrowing, then that's the problem. Don't try to match shapes, *share definitions*.

Do not name variant types but leave the discriminant values as bare string literals duplicated between each type definition and its construction site. Example: `result: 'skipped' in the type and result: 'skipped' in the factory` only match by shape; there's no shared definition. Find every site, then bind each discriminant to a single named definition that both the type (`typeof`) and the value reference.

## Derived-Boolean Elimination

Don't carry derived booleans in data shapes. A boolean field whose value is computed from other fields in the same value, and whose only consumer use is to gate iteration of those other fields or as a test-API signal, is not real information — it is a derivation that has leaked into the schema. The fix is to remove the boolean and have consumers compute it at the point of use. For optional single-instance data, use `T | null` instead of `{ flag: boolean; data: T }` pairing. For collections, an empty array IS "no items"; no boolean needed. NOOP sentinel objects exist only when a shape forces an entry to be present where semantically there should be none; remove the forcing constraint and the sentinel disappears. Consumers iterate the writes directly; "did anything change" is `write !== null` or `writes.length > 0` at the point of use, not a stored flag.

## Bun-First Testing Specifics

- Prefer Bun-native APIs and Bun-first patterns when implementing tests, especially for mocking and file access.
- For persistence-sensitive assertions, mock the specific real invocation with `spyOn` at the global call site (`spyOn(Bun, 'write')`, `spyOn(Bun, 'spawn')`, `spyOn(globalThis, 'fetch')`) — built by a factory, installed/restored through a parent-`describe` `beforeEach`/`afterEach(mock.restore)` lifecycle. Do not build an injected boundary, and do not reach for `mock.module('bun', …)` (it does not intercept Bun's built-in module). Filesystem reads may stay real over a per-test `mkdtemp` dir.
- Capture terminal output by spying the specific `console` method (`spyOn(console, 'log' | 'warn' | 'error')`) and asserting the recorded calls with `toHaveBeenCalledWith(expect.stringContaining(…))`; the spy patches the real `console` in place, so host-effect tests stay serial.
- Use `bun:test` and keep tests close to the behavior they defend. Check for existing helpers before creating new ones.
